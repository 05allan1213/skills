# AI 编程约束协议 — Go 后端开发通用版

> 本文档约束 AI 在参与 Go 后端项目开发时的行为规范。  
> 本版目标是让 AI 按照真实 Go 后端工程流程推进：先读上下文、明确边界、小步实施、及时验证、同步文档，而不是只做表面修补或凭空改代码。

---

## 0. 全局铁律

1. **严格以当前仓库代码、需求文档和用户本轮明确目标为准实施，不得脱离上下文自行假设。**
2. **不得在未完整读取相关文件上下文的情况下修改代码。**
3. **在不引入新依赖、不破坏现有接口语义的前提下，允许按需求对模型、API、持久化、配置、测试和文档做必要扩展。**
4. **优先打通可验证主链路：参数可接收、逻辑可执行、数据可保存、接口可查询、行为可验证；长期治理项可分阶段收敛，但不能阻塞主链路实现。**
5. **每次对话只解决一个明确子任务；修改前必须完整读取相关文件上下文。**
6. **每完成一个逻辑闭环，必须提供可执行的验证方式；未经验证的改动视为未完成。**
   - 验证说明默认**优先给出 `curl` 联调命令**，自动化测试与编译命令作为补充。
   - 若用户未另行指定地址，联调命令默认按 `http://127.0.0.1:8080/api/v1` 编写。
7. **每完成一步并确认无误后，必须同步更新对应需求文档、接口文档或说明文档。**
8. **每完成一个小块后，必须同步维护仓库根目录的 `how.md`，用面试可复述的口径说明“这个小块要解决什么问题、怎么实现、为什么这么做、结果是什么”。**
9. **`how.md` 不要求写成详细设计文档，但必须保证后续能据此清楚讲出该需求的实现路径。**
10. **git 提交只包含本次真正修改的代码文件；文档文件不进入代码提交命令。**
11. **禁止引入新的第三方依赖，除非用户明确确认。**

---

## 1. 当前阶段固定边界

以下边界在当前阶段视为默认约束，不得擅自偏离：

1. 对外接口字段、响应结构和错误语义默认保持兼容，除非用户明确要求修改。
2. Go 后端项目应优先落地主链路，不能只停留在局部字段修补或临时兼容。
3. 允许在当前服务内新增必要的结构体、持久化字段、接口和 service 逻辑，但必须服务于本轮明确需求。
4. 运行态数据应以数据库或当前项目既有权威来源为准，不能继续依赖临时默认值、历史模板或硬编码逻辑作为真实来源。
5. 无法由当前服务权威判定的校验，例如外部系统状态、第三方服务结果、部署环境差异等，允许保留下沉到对应系统最终裁决。
6. 本期优先打通：
   - 接口查询；
   - 接口更新；
   - 参数校验；
   - 业务逻辑主链路；
   - 数据持久化；
   - 状态联动；
   - 基础测试与联调验证。
7. 以下重治理项允许后置，但必须在文档里明确列为“未完成”而不是被假装已解决：
   - 完整审计链路；
   - 完整前端页面承接；
   - 外部系统端到端验收；
   - 历史数据全量治理；
   - 完整监控与告警；
   - 大规模重构。

## 2. 需求确认拦截规则

### 2.1 通用规则

实施任何代码前，必须先检查当前需求是否已经具备明确结论。

需要确认的内容包括：

1. 本轮目标是否明确；
2. 是否涉及接口变更；
3. 是否涉及数据库变更；
4. 是否涉及配置变更；
5. 是否涉及权限、鉴权或安全逻辑；
6. 是否涉及兼容旧数据；
7. 是否允许引入新依赖；
8. 是否需要同步文档；
9. 是否需要补充测试；
10. 是否存在用户未确认的设计分歧。

- 若已有确认：按确认内容实施。
- 若没有确认：立即停下，向用户说明涉及的条目、当前可选方案和建议默认值，等待确认后再继续编码。
- 若用户明确要求“按你判断来”：AI 可以基于当前代码风格和最小必要改动推进，但必须说明假设。

### 2.2 当前默认可直接实施的关键结论

1. 默认不引入新的第三方依赖。
2. 默认不改变已有对外接口主字段。
3. 默认不删除已有兼容逻辑。
4. 默认不进行大范围重构。
5. 默认不修改安全、权限、鉴权逻辑，除非本轮目标明确涉及。
6. 默认优先复用现有项目分层，例如 handler、service、repository、model。
7. 默认 Go 代码必须执行 `gofmt`。
8. 默认错误必须显式处理，不得吞错。
9. 默认新增或修改 HTTP API 后必须提供 `curl` 验证命令。
10. 默认涉及核心业务逻辑时需要补充单元测试或说明无法测试的原因。
11. 默认文档与代码需要同步更新。
12. 默认提交命令只包含本轮真正修改的代码文件。

### 2.3 实施前自检格式

每个代码步骤开始前，AI 必须输出：

```text
[开发检查] 步骤 X.X：<步骤名>
  本轮目标：<本步骤要解决的问题>
  涉及文件：<预计相关文件>
  是否涉及接口变更：是 / 否
  是否涉及数据结构变更：是 / 否
  是否涉及配置变更：是 / 否
  是否涉及数据库变更：是 / 否
  是否需要测试：是 / 否
  是否需要文档同步：是 / 否
  当前状态：可实施 / 未确认 — 停下询问
```

## 3. 实施策略

### 3.1 允许做的事情

1. 在现有接口、service、repository、model 链路中按需求补充必要逻辑。
2. 在现有数据模型上新增必要字段，但必须先确认是否涉及数据库迁移。
3. 新增结构化持久化模型，并提供查询/更新 API。
4. 在服务内实现基础参数校验、去重、排序、枚举限制、边界限制与全量替换语义。
5. 改造现有下游调用逻辑，使运行态配置或数据库记录成为真实来源。
6. 在创建、更新、删除、状态变更等场景下复用现有链路触发必要联动。
7. 补充必要的单元测试、接口联调说明和文档说明。
8. 在不改变外部语义的前提下，修复明显 bug、编译错误、测试失败和格式问题。
9. 按现有项目风格补充错误处理、日志、参数校验和返回值处理。
10. 对无法本轮完成的部分，在文档中明确列为未完成或依赖外部系统。

### 3.2 当前阶段禁止做的事情

1. 不得在用户未确认的情况下引入新第三方依赖。
2. 不得擅自改变已有对外主字段或接口语义。
3. 不得伪造“端到端已经完全闭环”，如果仍依赖前端、运维、外部服务或人工验证，必须明确写出。
4. 不得为了追求“最小改动”而故意跳过主需求。
5. 不得在没有文档同步的情况下私自改变已确认接口语义。
6. 不得不读上下文直接改代码。
7. 不得编造不存在的函数、文件、接口或测试结果。
8. 不得吞掉错误或用 `panic` 处理普通业务错误。
9. 不得在生产代码中留下临时调试输出。
10. 不得进行与本轮目标无关的大规模重构。

## 4. 推荐实施顺序

当前阶段按以下顺序推进：

1. 先确认需求、边界和相关文档是否已经明确。
2. 先完整阅读相关代码上下文。
3. 先落地必要的数据结构、参数结构或模型调整。
4. 再补充 service 主业务逻辑。
5. 再补充 handler、router 或对外入口。
6. 再补充 repository、DAO 或持久化逻辑。
7. 然后补充状态联动、异常语义和必要测试。
8. 最后同步文档状态、联调命令和提交命令。

---

## 5. 验证要求

每次完成代码后，必须至少提供以下一种验证：

1. **优先提供可直接执行的 `curl` 联调命令**，默认使用 `http://127.0.0.1:8080/api/v1`。
2. 相关单元测试命令。
3. 相关模块编译命令。
4. 明确的人工联调步骤。

### 5.1 `curl` 优先约束

后续所有验证说明默认遵循以下规则：

1. 先给 `curl` 命令，再给 `go test` / 编译命令。
2. `curl` 示例中的 URL 默认使用 `http://127.0.0.1:8080/api/v1`，除非用户明确说明已变更。
3. 若接口需要动态参数，优先提供带 `<id>`、`<user_id>`、`<resource_id>` 一类占位符的可复制命令，不要只写抽象步骤。
4. 若接口需要鉴权，`curl` 命令中的 token 默认写为 `Authorization: Bearer $TOKEN`，以便直接使用环境变量。
5. 每条 `curl` 命令后必须紧跟“正常预期结果”，至少说明应成功/应失败、预期响应码或业务码、以及需要关注的关键字段。
6. 涉及接口类需求的联调至少覆盖：
   - 查询接口；
   - 创建接口；
   - 更新接口；
   - 删除或关闭接口；
   - 异常参数；
   - 查询最终结果。

---

## 6. 文档同步要求

1. 只要实现行为与现有需求文档不一致，必须先改文档再改代码，或同一轮同步改完。
2. 若用户补充新的设计结论，必须先更新对应需求文档，再继续实施。
3. 每轮结束时必须把当前状态按以下四类同步到文档：
   - 已完成；
   - 当前后端还能继续做；
   - 必须等前端、运维或外部系统才能继续做；
   - 明确不在本期做。
4. 若本轮完成了新的“小块”，除更新需求文档外，还必须同步更新根目录 `how.md`，至少补齐：
   - 这个小块解决的问题；
   - 实际改了什么；
   - 为什么这样实现；
   - 最终达到了什么结果；
   - 如何验证。

---

## 7. 提交说明要求

当且仅当本轮包含代码修改时，最终回复中附带 git 提交命令，要求：

1. commit message 使用英文。
2. 只包含本轮真正修改的代码文件。
3. 不包含 `.md`、`.sh` 等非代码文件。
4. 若文档也需要提交，必须单独给出文档提交命令。
5. 不得把未修改文件、无关文件、临时文件加入提交。

示例：

```bash
git add internal/service/user.go internal/api/user_handler.go
git commit -m "Add user update validation"
```

如果需要单独提交文档：

```bash
git add docs/user_api.md how.md
git commit -m "Update user API documentation"
```
