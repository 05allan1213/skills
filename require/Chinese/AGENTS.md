# Go 项目 AI 开发公共约束

本文档用于约束 AI 在本 Go 项目中的代码实现行为。

核心目标：

- 小步开发
- 最小改动
- 可测试
- 可回滚
- 不盲目重构
- 不盲目加依赖
- 不覆盖用户已有修改
- 不假装测试通过

本文件只约束“怎么实现”，不规定“实现什么功能”。

---

## 1. 总原则

AI 在本项目中必须遵守以下原则：

1. 先理解代码，再提出方案，不直接上手修改。
2. 任何需求都应拆成小模块推进。
3. 每个小模块只解决一个明确问题。
4. 每个小模块完成后必须停下汇报。
5. 不做与当前任务无关的重构、格式化、改名、移动文件。
6. 不随意删除已有代码。
7. 不随意改变 API、配置、目录结构、公共函数签名。
8. 不盲目新增依赖。
9. 修改后必须执行必要验证。
10. 未执行测试或测试失败时必须明确说明。
11. 未经用户明确授权，不执行 git commit、git push、reset、clean 等危险操作。

---

## 2. 标准工作流程

每次开发任务按以下流程执行：

```text
理解代码
→ 判断最小修改范围
→ 拆分小模块
→ 说明本次准备改什么
→ 等待用户确认
→ 实现当前小模块
→ 格式化和测试
→ 汇报修改结果
→ 准备 git 提交建议
→ 等待用户确认是否继续
```

如果任务非常小，可以简化说明，但仍然必须明确：

```text
修改范围
涉及文件
验证方式
风险点
```

用户未确认前，不继续编码。

## 4. 任务拆分规则

拆分需求时遵守：

1. 一个模块只做一件事。
2. 一个模块尽量少改文件。
3. 一个模块完成后应能单独测试。
4. 一个模块应适合单独 git commit。
5. 不把功能开发、重构、依赖升级、格式化、文档大改混在一起。

错误示例：

```text
一次性完成 Redis 缓存、限流、WebSocket、告警推送、部署改造。
```

正确示例：

```text
1. 增加 Redis 客户端初始化
2. 封装缓存读写接口
3. 在一个查询接口中接入缓存
4. 补充缓存单元测试
5. 再考虑限流或 WebSocket
```

---

## 5. 代码修改约束

实现时必须遵守：

1. 只修改当前模块需要修改的文件。
2. 不顺手重构无关代码。
3. 不顺手改命名、改目录、改包结构。
4. 不删除看似无用但未确认用途的代码。
5. 不为了编译通过而注释掉核心逻辑。
6. 不留下临时代码、调试代码、无用 TODO。
7. 不生成大量无意义样板代码。
8. 不用复杂架构解决简单问题。
9. 不在 handler 中堆大量业务逻辑，必要时拆到 service/client/model，但不强行过度分层。

禁止随意修改：

```text
HTTP API 路径
请求参数
响应结构
配置字段
环境变量
Prometheus 指标名
Redis Key
K8s Service 名称
Helm values 结构
公共函数签名
包名
目录结构
```

如果必须修改，必须先说明原因、兼容方案和影响范围。

---

## 6. 依赖管理约束

新增依赖前必须先说明：

```text
新增依赖：
xxx

用途：
xxx

为什么标准库或已有依赖不能解决：
xxx

可能影响：
- go.mod/go.sum
- 镜像体积
- 安全风险
- 部署复杂度
```

未经确认，不得执行：

```bash
go get -u ./...
go get -u
go mod tidy
```

只有当前模块确实需要新增依赖时，才允许修改 `go.mod` / `go.sum`。

执行 `go mod tidy` 前必须说明原因。

## 7. 错误处理约束

禁止：

```go
_ = err
panic(err)
fmt.Println("debug")
log.Println("debug")
```

除非有明确理由并说明。

错误处理要求：

1. 错误要返回或记录。
2. 错误要补充必要上下文。
3. 对外响应不能泄露敏感信息。
4. 外部调用必须考虑 timeout。
5. 后台 goroutine 的错误不能丢失。
6. goroutine 中要避免 panic 导致服务异常退出。

推荐：

```go
if err != nil {
    return fmt.Errorf("load config: %w", err)
}
```

## 8. 配置约束

禁止硬编码：

```text
IP
端口
域名
账号
密码
Token
Namespace
镜像仓库
数据库地址
Redis 地址
Prometheus 地址
```

新增配置时必须说明：

```text
配置名：
默认值：
用途：
是否必填：
是否敏感：
本地如何配置：
Docker 如何配置：
K8s / Helm 如何配置：
```

敏感配置必须走 Secret 或环境变量，不得写死在代码、普通配置文件或 README 示例中。

## 9. 测试约束

新增或修改核心逻辑时，优先补测试。

优先测试：

```text
纯函数
参数校验
配置加载
请求解析
响应构造
错误处理
Redis Key 生成
PromQL 生成
缓存逻辑
限流逻辑
状态流转
```

测试优先级：

```text
1. 单元测试
2. 表驱动测试
3. 边界测试
4. 必要时再做集成测试
```

单元测试不得强依赖：

```text
真实 Redis
真实 MySQL
真实 Prometheus
真实 Kubernetes
公网服务
本机固定端口
```

应优先使用：

```text
interface
mock
httptest
临时目录
内存实现
```

## 10. 验证约束

每完成一个小模块，Go 项目默认执行：

```bash
goimports -w <本次修改的 Go 文件>
go test ./...
go vet ./...
```

如果全量测试不适合执行，必须说明原因，并执行最小可用测试：

```bash
go test ./指定包
```

如果修改了 Docker：

```bash
docker build
```

如果修改了 Helm / K8s：

```bash
helm lint
kubectl apply --dry-run=client
```

如果修改了 Prometheus 配置或规则：

```bash
promtool check config
promtool check rules
```

无法执行某个命令时，必须明确说明：

```text
未执行 xxx，原因是：xxx
```

不得使用以下说法冒充验证：

```text
应该没问题
理论上通过
看起来没问题
未测试但应该可以
```

必须明确区分：

```text
已执行并通过
已执行但失败
未执行
```

## 11. Git 安全约束

实现前建议检查：

```bash
git status
```

如果发现已有未提交修改，必须提醒用户确认，不能覆盖用户改动。

未经用户明确授权，禁止执行：

```bash
git commit
git push
git reset --hard
git checkout .
git clean -fd
```

每个模块完成后，只能给出建议提交命令：

```bash
git add <files>
git commit -m "feat: xxx"
```

推荐提交类型：

```text
feat: 新功能
fix: 修复问题
refactor: 不改变行为的重构
test: 测试
docs: 文档
chore: 工程配置
ci: CI/CD
style: 格式化
```

## 13. 禁止行为清单

除非用户明确授权，禁止：

```text
1. 一次性重写整个项目
2. 大范围删除代码
3. 擅自修改目录结构
4. 擅自升级大量依赖
5. 擅自引入新中间件
6. 擅自改变 API 响应结构
7. 擅自改变配置字段
8. 擅自修改公共函数签名
9. 擅自修改 Prometheus 指标名
10. 擅自执行 git commit
11. 擅自执行 git push
12. 擅自执行 git reset --hard
13. 擅自执行 git clean -fd
14. 擅自删除用户未提交代码
15. 为了测试通过而删除测试
16. 为了编译通过而注释核心逻辑
17. 留下伪代码冒充完成
18. 留下 TODO 冒充已完成
19. 假称测试通过但实际未执行
20. 用复杂架构掩盖简单问题
```

---

## 14. 最终原则

AI 的职责是辅助开发者安全推进需求，而不是替代开发者盲目写代码。

默认策略：

```text
先读代码
再拆任务
小步实现
及时验证
停下汇报
准备提交
等待确认
```

只要出现以下情况，必须停下来说明：

```text
需求不明确
代码上下文不足
改动影响范围过大
需要新增依赖
需要改变接口或配置
测试无法执行
存在用户未提交修改
存在兼容性风险
```
