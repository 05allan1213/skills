# Go Code Review Skill

本 Skill 用于 Go 项目的代码审查、diff 审查、PR 审查、AI 生成代码检查。

公共开发底线遵守项目根目录中的：

```text
AGENTS.md
CLAUDE.md
```

本 Skill 只补充“代码审查”场景下的额外规则。

---

## 1. 适用场景

当用户要求以下任务时，使用本 Skill：

```text
审查当前 git diff
审查某个 PR
检查 AI 生成代码是否靠谱
检查某次需求实现是否合理
检查 Bug 修复是否完整
检查测试是否充分
检查接口兼容性
检查配置、依赖、部署改动风险
```

---

## 2. Review 原则

代码审查必须遵守：

1. 先看需求目标，再看实现是否匹配。
2. 先看风险，再看风格。
3. 优先指出会导致错误、兼容性破坏、数据问题、线上风险的问题。
4. 不纠结无意义格式问题。
5. 不要求为了“更优雅”而大规模重构。
6. 每个问题必须说明原因、影响和建议修改方式。
7. 不确定的问题必须标注为“需要确认”，不能当成确定结论。
8. Review 结果要按优先级分级。

---

## 3. Review 前必须确认

开始审查前，先明确：

```text
审查目标：
xxx

审查范围：
- 文件：xxx
- diff：xxx
- PR：xxx
- 模块：xxx

重点关注：
- 功能正确性
- 兼容性
- 错误处理
- 测试充分性
- 并发安全
- 配置和部署风险
- 依赖风险
```

如果上下文不足，必须说明：

```text
当前只能基于已有 diff 审查，无法确认完整业务意图。
```

---

## 4. 问题分级

Review 问题按以下等级输出：

```text
P0：必须修改
会导致编译失败、测试失败、数据错误、panic、安全问题、接口严重不兼容、线上故障。

P1：建议修改
当前可能能跑，但有明显边界问题、错误处理不足、兼容性风险、测试缺失、维护成本较高。

P2：可选优化
代码可读性、局部结构、命名、轻微重复、注释补充等，不阻塞合入。
```

不要把所有问题都说成 P0。

---

## 5. 核心审查清单

必须重点检查：

```text
功能是否满足需求
是否引入无关改动
是否改动范围过大
是否破坏已有 API
是否改变配置字段
是否改变公共函数签名
是否新增依赖且理由充分
是否存在硬编码
是否吞错误
是否缺少 timeout
是否存在 goroutine 泄漏
是否存在并发安全问题
是否存在资源未关闭
是否存在敏感信息泄露
是否有足够测试
测试是否依赖不稳定外部服务
是否假装完成但留下 TODO / 伪代码
```

---

## 6. Go 代码重点审查项

Go 代码重点看：

```text
error 是否被正确处理
context 是否正确传递
defer close 是否必要且存在
HTTP client 是否设置 timeout
goroutine 是否可退出
channel 关闭所有权是否清晰
map 是否存在并发读写
slice / map / pointer 是否可能 panic
interface 抽象是否必要
包名和目录结构是否符合项目现状
命名是否与已有代码风格一致
```

---

## 7. API 审查项

如果涉及 HTTP API，必须检查：

```text
路径是否合理
方法是否合理
请求参数是否校验
错误响应是否一致
响应结构是否兼容
状态码是否正确
handler 是否堆积过多业务逻辑
是否泄露内部错误细节
是否有必要测试
```

---

## 8. 配置和部署审查项

如果涉及配置、Docker、K8s、Helm，必须检查：

```text
配置是否集中管理
默认值是否合理
敏感信息是否进入 Secret 或环境变量
values.yaml 是否泄露敏感信息
是否影响已有部署
是否需要迁移说明
是否提供启动和验证方式
```

---

## 9. 测试审查项

必须检查测试：

```text
是否覆盖核心逻辑
是否覆盖错误分支
是否覆盖边界情况
是否使用表驱动测试
是否依赖真实外部服务
是否为了通过而降低测试质量
是否删除了关键测试
是否只测 happy path
```

如果测试不足，应说明缺口和建议补充的 case。

---

## 10. 输出格式

Review 输出使用以下格式：

```text
审查结论：
- 通过 / 不建议合入 / 需要修改后再看

总体评价：
xxx

P0 必须修改：
1. 问题：xxx
   影响：xxx
   建议：xxx

P1 建议修改：
1. 问题：xxx
   影响：xxx
   建议：xxx

P2 可选优化：
1. 问题：xxx
   建议：xxx

测试建议：
- xxx

风险点：
- xxx

建议下一步：
xxx
```

如果没有发现明显问题，输出：

```text
审查结论：
- 暂未发现阻塞问题

仍需注意：
- xxx

建议验证：
- go test ./...
- go vet ./...
- 相关接口手动验证
```

---

## 11. 禁止行为

代码审查中禁止：

```text
1. 只说“看起来没问题”
2. 只给风格建议，不看功能风险
3. 不分优先级
4. 没有证据就断言问题
5. 要求无意义大重构
6. 忽略测试缺失
7. 忽略兼容性风险
8. 忽略配置和部署风险
9. 忽略用户已有修改
10. 把个人偏好包装成必须修改
```
